## Windows Subsystem for Linux (WSL)

<Callout type="info" title="MacOS users can skip this section">
MacOS is UNIX-based, so you can jump straight to installing [package managers](#install-package-managers).
</Callout>

Install [Windows Subsystem for Linux (WSL)](https://learn.microsoft.com/en-us/windows/wsl/install).

<Accordions>
<Accordion title="What's WSL?">
Introduced by Microsoft in 2016, WSL allows people to run a real Linux kernel inside Windows.

Before WSL, people had to:
- Use a virtual machine, slow.
- Dual boot on the same machine, tedious and risks bricking the machine.
- Replace their main OS with Linux altogether, scary but the best option.

[Learn more](https://en.wikipedia.org/wiki/Windows_Subsystem_for_Linux).
</Accordion>
<Accordion title="Why use WSL instead of switching to Linux?">
WSL allows you to:
- have the best of both worlds by running Linux and Windows simultaneously.
- Use Linux tools and commands directly from Windows without switching operating systems.
- Run various development environments smoothly without leaving your Windows OS.

You can still access the Windows filesystem from Linux and vice versa and avoid the hassle of dual booting.

[Why Dual booting on the same drive is not worth it](https://www.makeuseof.com/tag/risks-dual-booting-windows-linux-operating-systems/).
</Accordion>
  <Accordion title="But im not comfortable working from the shell">
    Although it is better working from the shell. WSL can run in the VS Code terminal, or any other
    IDE terminal. Rather than cloning the repo on your Windows filesystem, you'd be
    cloning it in a directory of your choice in the GNU/Linux OS.

    You can still access the windows filesystem from WSL and vice versa, browsing through the Linux
    directories with the Windows file explorer graphically.
  </Accordion>
</Accordions>

<Callout type="info" title="Why we use pnpm">
We use **pnpm** instead of **npm** because it is faster, more efficient with disk space, and enforces stricter dependency management.  
Unlike npm or yarn, pnpm uses a content-addressable store and hard links, which means packages are downloaded once and reused across all projects ‚Äî saving both time and storage.  
It also prevents phantom dependencies by ensuring that only explicitly listed packages are accessible, leading to more reliable builds.

### Set Your Git Identity (Required)

Git needs your name and email to attach to every commit you make. Without this, you'll see errors like `"Author identity unknown"` or `"Please tell me who you are"` when you try to commit‚Äîespecially if you're using [lazygit](https://github.com/jesseduffield/lazygit).

Run these commands to set your identity globally:

```sh title="Terminal"
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```

Verify it worked:

```sh title="Terminal"
git config --global --get user.name
git config --global --get user.email
```

<Callout type="info" title="Privacy tip">
If you'd rather not expose your real email, you can use GitHub's private "noreply" email address. Find it in your [GitHub email settings](https://github.com/settings/emails) under "Keep my email addresses private".
</Callout>
</Step>

### GitHub SSH Setup (Recommended for Pushing)

If you plan to push code using `git push` or lazygit (not just `gh` CLI), setting up SSH keys prevents `Permission denied (publickey)` errors.

<Callout type="info" title="Already using gh auth?">
`gh auth login` handles authentication for the GitHub CLI, but SSH keys are still needed when pushing via regular git commands or lazygit with an SSH remote.
</Callout>

**1. Generate an SSH key:**

```sh title="Terminal"
ssh-keygen -t ed25519 -C "your_email@example.com"
```

Press Enter to accept the default location (`~/.ssh/id_ed25519`). You can set a passphrase or leave it empty.

<Callout type="warn" title="Keep the default filename">
Don't rename the key file unless you know how to configure SSH to find it. Stick with `id_ed25519`.
</Callout>

**2. Start the SSH agent and add your key:**

```sh title="Terminal"
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
```

**3. Copy your public key:**

```sh title="Terminal"
cat ~/.ssh/id_ed25519.pub
```

Copy the entire output.

**4. Add the key to GitHub:**

Go to **GitHub ‚Üí Settings ‚Üí SSH and GPG keys ‚Üí New SSH key**. Paste your public key and save.

![Image](/images/addsshkey.png)

**5. Test your connection:**

```sh title="Terminal"
ssh -T git@github.com
```

You should see a message like "Hi username! You've successfully authenticated..."
![Image](/images/lazygit-ssh.png)


**6. Check your remote URL:**

```sh title="Terminal"
git remote -v
```

If it shows `https://github.com/...` and you want to use SSH, switch it:

```sh title="Terminal"
git remote set-url origin git@github.com:OWNER/REPO.git
```

Replace `OWNER/REPO` with your fork (e.g., `yourusername/2026`).
</Step>

## Verify Your Setup

Before diving in, run these quick checks to make sure everything is configured:

```sh title="Terminal"
# Git identity set?
git config --global --get user.name
git config --global --get user.email

# SSH key exists?
ls -la ~/.ssh

# SSH agent has your key?
ssh-add -l

# GitHub connection works?
ssh -T git@github.com

# lazygit installed?
lazygit --version
```

If any of these fail, revisit the relevant section above.

## Troubleshooting

<Accordions>
<Accordion title="Permission denied (publickey)">
This means Git can't authenticate with GitHub over SSH. Common fixes:

1. **Check if your key is loaded:**
   ```sh
   ssh-add -l
   ```
   If it says "no identities", add your key:
   ```sh
   eval "$(ssh-agent -s)"
   ssh-add ~/.ssh/id_ed25519
   ```

2. **Verify GitHub has your public key:**
   ```sh
   cat ~/.ssh/id_ed25519.pub
   ```
   Compare with what's in GitHub ‚Üí Settings ‚Üí SSH and GPG keys.

3. **Test the connection:**
   ```sh
   ssh -T git@github.com
   ```

4. **Check your remote uses SSH (not HTTPS):**
   ```sh
   git remote -v
   ```
   If it shows `https://`, switch to SSH:
   ```sh
   git remote set-url origin git@github.com:OWNER/REPO.git
   ```
</Accordion>

<Accordion title="src refspec main does not match any">
This usually means you're trying to push but there's no commit yet, or the branch name doesn't match.

1. **Check your current state:**
   ```sh
   git status
   git branch
   ```

2. **If there's no commit yet, create one:**
   ```sh
   git add .
   git commit -m "chore: initial commit"
   ```

3. **Push to the correct branch:**
   ```sh
   git push -u origin main
   ```

4. **If your branch is called `master` but remote expects `main`:**
   ```sh
   git branch -M main
   git push -u origin main
   ```
</Accordion>

<Accordion title="Author identity unknown">
Git doesn't know who you are. Set your identity:
```sh
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```
See [Set Your Git Identity](#git-identity) for details.
</Accordion>
</Accordions>
